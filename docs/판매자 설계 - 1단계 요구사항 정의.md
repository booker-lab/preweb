## 1. 페르소나 및 목적

**누가 (Who)**
신선 화훼·농산물을 직접 소싱하여 소비자에게 판매하는 개인 사업자 또는 소규모 법인.
MVP: 디어 오키드 단독 운영자. 이후 다중 판매자 입점 구조로 확장.

**무엇을 (What)**
상품 등록, 주문 접수·처리, 배송 운영, 정산 확인을 하나의 모바일 중심 앱에서 수행.

**왜 (Why)**
소비자 앱과 실시간으로 연동된 주문 현황을 현장(꽃차 운행 중 포함)에서 즉시 처리하고,
매출·정산 데이터를 투명하게 관리하여 사업 운영 효율을 높이기 위함.

## 2. 사용자 스토리

| # | 스토리 | 연결 기능 |
|---|--------|----------|
| S1 | 나는 카카오 또는 이메일로 간편하게 판매자로 가입하고, 사업자 정보를 등록하여 신뢰할 수 있는 소싱처임을 증명하고 싶다. | 판매자 온보딩 |
| S2 | 나는 새 주문이 들어오면 즉시 알림을 받고, 주문을 확인한 뒤 준비 완료 예정 시간을 설정하고 싶다. | 주문 관리, 준비 예약 |
| S3 | 나는 상품을 빠르게 등록하고, 일반 판매와 공동구매 방식을 자유롭게 전환하고 싶다. | 상품 관리 |
| S4 | 나는 오늘의 매출 합계, 기간별 실적, 개별 주문 정산금을 한눈에 확인하여 자금을 관리하고 싶다. | 정산/매출 관리 |
| S5 | 나는 꽃차 직배송을 위한 거점(Hub) 정보를 등록하고, 소비자의 픽업 현황을 확인하고 싶다. | 거점 관리 |
| S6 | 나는 혹한기·혹서기 기상 특보 시 전국 택배 주문을 즉시 차단하고 싶다. | 배송 제어 |
| S7 | 나는 꽃차 운행 중 모바일에서 주문 처리가 불편하지 않아야 하고, PC에서도 동일하게 사용할 수 있어야 한다. | 반응형 PWA |

## 3. 핵심 기능 목록

### ✅ Must Have (필수 구현)

---

#### 판매자 온보딩

- 로그인: **이메일 + 카카오 OAuth** (소비자 앱과 동일한 `/auth/login`, `/auth/register` 엔드포인트 공유, `role: 'seller'` 명시)
- 가입 후 최초 1회 **사업자 프로필 설정** 필수:
  - 필수: 상호명, 대표자명, 연락처, 소재지
  - 선택: 사업자등록번호, 로고 이미지 (미입력 시에도 `active` 전환 가능)
- MVP: 관리자(플랫폼 운영자)가 `stores` 문서를 생성한 뒤 판매자에게 `storeId` 공유 → 판매자가 프로필을 채우는 방식
- Phase 2: 판매자 자체 가입 → 플랫폼 승인 플로우 추가

> **설계 의도**: 멀티앱 구조상 `/auth/register`에 `role: 'seller'`를 명시해야 한다. (`docs/specs/auth.md` §4 참조)

---

#### 주문 관리 대시보드

- Firestore 실시간 리스너 기반 주문 목록 (`GET /stores/:storeId/orders`)
- 상태별 탭 필터: **처리 필요** (ACCEPTED, CONFIRMED) · **준비 중** (PREPARING) · **배송 중** (DELIVERING, HUB_ARRIVED) · **완료** (DELIVERED, PICKED_UP, REVIEWED) · **취소** (CANCELLED)
- 상태 전환 버튼:
  - `ACCEPTED / CONFIRMED → PREPARING`: "준비 시작" 버튼
  - `* → CANCELLED`: "강제 취소" 버튼 (사유 입력 필수)
- **준비 완료 예약**: PREPARING 전환 시 "드라이버 수거 예정 시간" 입력 (선택)
  - `orders.preparedAt: Timestamp | null` 필드 추가 필요
  - 드라이버 앱의 수거 목록에 해당 시간이 표시됨

> **API 현황**: `PATCH /stores/:storeId/orders/:orderId/status` 구현 완료.
> 단, `preparedAt` 필드는 DB·API에 추가 필요.

---

#### 상품 관리

- 상품 목록 조회 (`GET /stores/:storeId/products`)
- 상품 등록 (`POST /stores/:storeId/products`):
  - 상품명, 가격, 카테고리(절화/난/관엽), 색상(멀티), 배송 사이즈(small/medium/large)
  - 판매 방식 선택: 일반 판매 / 공동구매
  - 공동구매 설정: 최소·최대 인원, 모집 마감일, 배송 예정일, 배송 수단, 할인 배송비
  - 이미지 업로드 (Firebase Storage → URL 저장)
- 상품 수정 · 활성/비활성 토글 (`PATCH /stores/:storeId/products/:id`, `/active`)
- 상품 삭제 (`DELETE /stores/:storeId/products/:id`)

> **API 현황**: 전체 구현 완료. 이미지 업로드만 클라이언트(Firebase Storage SDK) 처리 필요.

---

#### Daily Cap 관리

- 날짜별 최대 배송 슬롯 설정 (`PATCH /stores/:storeId/daily-caps/:date`)
- 날짜별 잔여 슬롯 조회 (`GET /stores/:storeId/daily-caps?from=&to=`)
- 소비자 결제 화면 Date Picker와 실시간 연동

> **API 현황**: 구현 완료.

---

#### 배송 설정 및 기상 제한 컨트롤러

- 배송비 설정 (`PATCH /stores/:storeId/delivery-config`):
  - 직배송·거점·택배 기본 배송비 및 무료 배송 기준
- **기상 제한 On/Off 토글**: `weatherRestrictionActive: boolean`
  - 활성화 시 소비자 결제 화면의 택배 옵션 자동 비활성화
  - 직배송·거점 픽업은 영향 없음

> **API 현황**: 구현 완료.

---

#### 정산 / 매출 관리 ← 신규 설계

**정산 계산식**

```
정산금 = totalAmount × (1 - commissionRate)

MVP:  commissionRate = 0  →  정산금 = totalAmount
확장: stores.commissionRate 필드로 판매자별 요율 관리 (코드 변경 없음)
```

**정산 확정 시점**: 주문 상태가 `REVIEWED` 또는 `DELIVERED` / `PICKED_UP` 도달 시
→ `settlements` 컬렉션에 정산 레코드 자동 생성 (NestJS 이벤트 기반)

**화면 구성**

| 뷰 | 내용 |
|----|------|
| 일별 요약 | 금일 완료 주문 건수 · 총 매출 · 정산 예정액 |
| 기간별 조회 | 달력 필터 → 기간 합계 (총 매출 / 수수료 / 정산금) |
| 주문별 상세 | 개별 주문: 상품가 + 배송비 - 수수료 = 정산금 |

**Should Have**: 기간별 CSV 다운로드

> **API 현황**: 미구현. `settlements` 컬렉션 및 관련 엔드포인트 신규 개발 필요.

---

#### 거점(Hub) 관리 ← 신규 설계

- 거점 등록·수정·삭제 (`hubs` 컬렉션 신규)
- 거점별 픽업 대기 주문 목록 (상태: `HUB_ARRIVED`)
- **픽업 인증 확인**: 소비자 픽업 코드 입력 → `PATCH /stores/:storeId/orders/:orderId/pickup-confirm`
  - 바코드 스캔 또는 6자리 코드 수동 입력

> **API 현황**: `pickup-confirm` 엔드포인트 구현 완료. 거점 CRUD는 신규 개발 필요.

#### 실시간 운영 알림

판매자에게 발송되는 알림 (**2026-03-28 확정** — 매 건 알림 과잉으로 정리):

| 코드 | 트리거 | 내용 |
|------|--------|------|
| `SELLER_GROUP_CONFIRMED` | 공동구매 목표 달성 (RECRUITING → CONFIRMED) | "OO 공동구매 목표 달성! 준비를 시작하세요." |
| `SELLER_GROUP_CANCELLED_LACK` | 미달 자동 취소 (RECRUITING → CANCELLED) | "OO 공동구매 미달 자동 취소 및 환불이 완료되었습니다." |
| `SELLER_ORDER_BATCH` | NestJS @Cron 스케줄러 (발송 시각 미결) | "오늘 N건 주문, 총 OOO원" (0건 시 미발송) |

> **제거된 알림**: `SELLER_NEW_ORDER`(일반 판매 매 건), `SELLER_ORDER_CANCELLED`(소비자 개인 취소) — seller 앱 스캐폴딩 시 코드에서도 제거.
> **배치 알림 발송 시각**: seller 앱 착수 시 확정 (1일 1~2회 옵션 검토 중)

## 4. 비기능 요구사항

### 반응형 (모바일 + 데스크탑)
- **모바일 우선 PWA**: 꽃차 운행 중 한 손 조작 가능한 UI
- **동일 계정으로 PC 브라우저 접속 가능**: Next.js 반응형 breakpoint로 레이아웃 분기
- 별도 URL: `seller.greenhub.kr` (Vercel 별도 배포)

### 정확성
- 정산 금액은 원(₩) 단위까지 오차 없이 계산
- 결제 취소·환불 건 즉시 정산 레코드에 반영

### 실시간성
- 주문 상태 변경: Firestore 실시간 리스너 (소비자 앱과 동일 채널)
- 알림 발송: 상태 전환 후 1초 이내

### 보안
- `storeId` 소유권 검증: 모든 API 요청에서 `JWT.storeId === path.storeId` 강제 (현재 orders에 누락 — seller 앱 착수 전 수정 필요)
- 픽업 코드: 6자리 숫자, 서버 생성, 단방향 검증

## 6. MVP 범위 및 다중 판매자 확장 계획

### MVP (현재)
```
- 단일 판매자: 디어 오키드
- 플랫폼 운영자가 stores 문서 생성 후 storeId 공유
- commissionRate = 0 (수수료 없음)
- 판매자 앱 로그인 후 자신의 storeId 데이터만 접근
```

### Phase 2 (다중 판매자)
```
추가 개발 항목:
  - 판매자 자체 가입 → 플랫폼 승인 플로우
  - stores.commissionRate 필드 값 설정 UI (운영자 어드민)
  - 소비자 앱: 가게 목록·가게 상세 페이지 추가
  - 정산 시스템: 판매자별 수익 분리·지급

변경 불필요 항목 (처음부터 storeId 구조로 설계됨):
  - DB 스키마, API 엔드포인트, 주문 처리 로직
  - 정산 계산 로직 (commissionRate = 0 → 실제 요율로 값만 변경)
```

## 7. DB 스키마 추가 정의

> 소비자 설계 1단계의 기존 스키마(stores, products, orders, dailyCaps, deliveryFeeConfig)에 추가되는 항목

### stores (확장)

```ts
stores {
  id: string
  name: string               // 상호명
  ownerId: string            // users 참조

  // 판매자 앱에서 추가 입력
  businessNumber: string     // 사업자 등록번호
  ceoName: string            // 대표자명
  phone: string
  address: string            // 소재지
  logoUrl: string | null     // Firebase Storage URL

  // 정산 설정
  commissionRate: number     // 0.0 ~ 1.0 (MVP: 0)

  createdAt: Timestamp
  updatedAt: Timestamp
}
```

### orders (필드 추가)

```ts
orders {
  // 기존 필드 유지 (소비자 설계 1단계 스키마 참조)
  // ...

  // 신규 추가
  preparedAt: Timestamp | null  // 드라이버 수거 예정 시간 (판매자가 PREPARING 전환 시 선택 입력)
}
```

### settlements (신규)

```ts
settlements {
  id: string                   // uuid
  storeId: string              // stores 참조
  orderId: string              // orders 참조
  userId: string               // 구매자 ID

  // 금액 내역
  totalAmount: number          // 소비자 결제 금액 (상품가 + 배송비)
  commissionRate: number       // 확정 시점 요율 스냅샷 (나중에 요율 변경돼도 불변)
  commissionAmount: number     // totalAmount × commissionRate
  settlementAmount: number     // totalAmount - commissionAmount

  status: 'pending' | 'confirmed' | 'paid' | 'cancelled'
  // pending:   ACCEPTED/RECRUITING ~ DELIVERED/REVIEWED
  // confirmed: REVIEWED 또는 DELIVERED/PICKED_UP 도달 시 — 판매자 지급 대기
  // paid:      운영자가 판매자 계좌로 이체 완료 (admin 화면에서 처리)
  // cancelled: 주문 CANCELLED 시 즉시 반영

  confirmedAt: Timestamp | null
  createdAt: Timestamp
  updatedAt: Timestamp
}
```

### hubs (신규)

```ts
hubs {
  id: string
  storeId: string              // stores 참조
  name: string                 // 거점명 (예: "서울 강남 거점")
  address: string
  addressDetail: string
  zipCode: string
  managerPhone: string | null  // 거점 담당자 연락처
  isActive: boolean
  createdAt: Timestamp
  updatedAt: Timestamp
}
```
