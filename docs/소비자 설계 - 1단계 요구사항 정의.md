## 1. 페르소나 및 목적

**누가 (Who)**
신선한 화훼 및 농산물을 합리적인 가격에 구매하고자 하는 개인 소비자 및 소규모 사업자.

**무엇을 (What)**
실시간 산지 시세를 반영한 상품 탐색 및 주문, 그리고 상품의 준비부터 배송까지의 전 과정을 실시간으로 확인하는 서비스.

**왜 (Why)**
유통 단계를 줄여 산지의 데이터가 가격에 즉각 반영되는 합리성을 누리고, 중간 유통 없는 빠른 조달 과정을 당근마켓과 같은 직관적인 거래 경험으로 확인하기 위함.

## 2. 사용자 스토리

| # | 스토리 | 연결 기능 |
|---|--------|----------|
| S1 | 나는 오늘의 산지 시세를 반영한 합리적인 가격을 확인하고, 투명하게 공개된 상품 정보를 바탕으로 원하는 상품을 검색하고 싶다. | 실시간 시세 카탈로그 |
| S2 | 나는 마음에 드는 상품을 장바구니에 담아 네이버페이나 카카오페이로 간편하게 결제하고 싶다. 공동구매 상품은 바로 참여하여 결제할 수 있어야 한다. | 장바구니, 간편 결제, 공동구매 즉시 결제 |
| S3 | 나는 내가 주문한 상품이 현재 '산지 확보'되었는지, '배송 중'인지 당근마켓 거래 피드처럼 실시간으로 확인하며 시스템의 속도감을 느끼고 싶다. | 실시간 주문 현황 보드 |
| S4 | 나는 앱에 접속하지 않아도 카카오톡이나 푸시 알림을 통해 내 주문이 어느 단계에 있는지 즉시 안내받고 싶다. | 멀티채널 알림 |
| S5 | 나는 앱스토어 설치 없이 홈 화면에 추가하여 필요할 때마다 앱처럼 빠르게 접속하고 알림을 받고 싶다. | PWA 설치 경험 |
| S6 | 나는 소셜 계정이 없어도 이메일로 가입하여 주문할 수 있어야 한다. | 이메일 로그인 |
| S7 | 나는 상품 구매 시 최소 인원이 모여야 진행되는 공동구매 방식으로 참여하고, 현재 몇 명이 참여 중인지 실시간으로 확인하고 싶다. 목표 인원이 채워지면 주문이 확정되고, 미달 시 자동 환불되는 구조가 신뢰할 수 있다. | 공동구매(모집형) 상품, 실시간 참여 현황 |

## 3. 핵심 기능 목록

### ✅ Must Have (필수 구현)

**인증**
- 소셜 로그인: NextAuth.js 기반 카카오, 네이버 (S6)
- 이메일 로그인: 이메일/비밀번호 기반 회원가입 및 로그인 (S6)
- 비회원 주문: 미지원 (모든 주문은 회원 로그인 필수)

**상품**
- 실시간 시세 카탈로그: 산지 시세 기반 상품 목록 및 상세 페이지 (S1)
  - 데이터 투명성과 조달 정보 중심의 UI
  - 상품 판매 방식: 일반 판매 / 공동구매(모집형) 선택 (판매자 앱에서 설정)
- 색상 필터: 카테고리 페이지에서 색상 기반 상품 탐색
  - 적용 대상: 절화 / 난 / 관엽 전체 (모든 카테고리에 색상 의미 있음)
  - 색상 옵션: 레드, 핑크, 화이트, 옐로우, 오렌지, 퍼플, 블루, 그린, 무늬, 브라운, 베이지, 블랙, 그레이
  - 멀티 선택 가능 (예: 레드 + 핑크 동시 선택)
  - 카테고리 탭 필터와 AND 조건으로 조합 (예: 절화 + 레드)
  - 판매자 앱 상품 등록 시 색상 멀티 선택 필드 필수 입력

**공동구매 (모집형) 상품** (S7)
- 상품 카드 및 상세에 실시간 참여 인원 현황 표시 (Firestore 실시간 리스너)
  - 예: `██████░░░░  18/30명 참여중 · 마감까지 1일 12시간`
- 최소 인원 달성 시 → 주문 자동 확정 (CONFIRMED)
- 마감 기한까지 최소 인원 미달 시 → 전체 취소 + Portone 자동 환불
- 결제 방식: 참여 시 즉시 결제 (방식 A 확정)
  - 미달 취소 시 Portone 환불 API로 자동 처리
- DB 스키마: 하단 'DB 스키마 정의 (전체)' 섹션 참조

**주문 & 결제**
- 장바구니 및 주문 생성 (S2)
- 배송 희망일 선택: 결제 화면 내 Date Picker로 배송 가능한 날짜 선택 (S2)
  - 일별 최대 수용량(Daily Cap) 초과 시 해당 날짜 비활성화 처리
  - Daily Cap은 판매자 앱에서 설정
  - **거점 픽업(Model B)도 동일한 Daily Cap 적용**: MVP 단계 인력 부족 상황을 고려하여 거점 픽업 주문도 날짜별 수용량 제한 관리
- 간편 결제: Portone SDK를 통한 카카오페이, 네이버페이, 카드 결제 연동 (S2)
  - 무통장 입금 미지원 (즉시 결제만 지원)
- 배송 수단 선택: 결제 화면에서 주소 기반 isMetropolitan 판별 후 사용 가능한 배송 수단 표시
  - 서울/경기: 꽃차 직배송 / 거점 픽업 / 택배 중 선택
  - 기타 지역: 택배만 표시
- 배송비 자동 계산: 배송 수단 + 품목 사이즈(deliverySize) 조합으로 결제 화면에서 자동 계산 표시
  - 공동구매 배송비: 일반 배송비 대비 할인 배송비 자동 적용 (판매자가 상품 등록 시 설정)
    - 이벤트/프로모션 시 판매자가 무료 배송으로 설정 가능 → 소비자 참여 유도 및 판매 촉진 도구로 활용
- 기상 조건 택배 제한: 판매자 앱에서 기상 제한 모드 활성화 시 소비자 결제 화면의 택배 옵션 자동 비활성화
  - 비활성화 시 안내 문구: "현재 기온 조건으로 인해 택배 배송이 일시 중단되었습니다"
  - 꽃차 직배송은 기상 무관 정상 운영

**실시간 추적**
- 실시간 주문 현황 보드: Firestore 실시간 리스너 기반 주문 상태 추적 UI (S3)
  - 당근마켓 스타일의 거래 피드 형태
  - 주문 상태 정의 (소비자 노출 기준):

| 상태 코드 | 소비자 노출 명칭 | 정의 및 전환 조건 | 적용 판매 방식 |
|-----------|----------------|-----------------|--------------|
| RECRUITING | 모집 중 | **공동구매 첫 상태.** 참여 결제 완료, 최소 인원 모집 중 | 공동구매 |
| CONFIRMED | 주문 확정 | 최소 인원 달성으로 공동구매 확정 | 공동구매 |
| ACCEPTED | 결제 완료 | **일반 판매 첫 상태.** 결제 확인 및 주문 접수 | 일반 판매 |
| PREPARING | 상품 준비 중 | 생산자 확인 및 포장 단계 | 공통 |
| DELIVERING | 배송 중 | 드라이버 픽업 완료 후 목적지(소비자 주소 또는 거점)로 이동 중 | 공통 |
| HUB_ARRIVED | 거점 도착 | 상품이 거점(협력 업체)에 도착, 소비자 픽업 대기 중 | 거점 픽업 |
| PICKED_UP | 픽업 완료 | 소비자가 픽업 코드 제시 후 상품 수령 완료. 거점 픽업의 배송 완료에 해당 | 거점 픽업 |
| DELIVERED | 배송 완료 | 수령 완료 (직배송·택배 전용) | 직배송·택배 |
| CANCELLED | 주문 취소 | 소비자에게 "[목표 미달성 취소]" 등 사유 명시 워딩 사용, 카카오 알림톡으로 즉시 사유 전달 + 자동 환불 | 공통 |
| REVIEWED | 구매 확정 | 리뷰 작성 완료 또는 구매 확정 버튼 클릭 시 (DELIVERED 또는 PICKED_UP 이후) | 공통 |

**알림**
- 카카오 알림톡: 주문 상태 변경 시 카카오톡 공식 알림 발송 (S4)
  - 알리고, 솔라피 등 비즈메시지 API 연동 (건당 약 8~9원)
  - 공동구매 전용 알림 추가:
    - 참여 완료: "OO 상품 모집에 참여하셨습니다. 현재 18/30명"
    - 마감 임박: "마감 2시간 전, N명이 더 필요해요. 주변에 공유해보세요!"
    - 목표 달성: "목표 인원이 모였습니다! 주문이 확정되었습니다."
    - 목표 미달 취소: "[목표 수량 미달성으로 취소] 결제 금액은 3~5일 내 환불됩니다."

**PWA**
- PWA 설치 경험 (S5)
  - manifest.json 설정 (앱 이름, 아이콘, 테마 색상)
  - Service Worker 등록
  - A2HS(Add to Home Screen): 자동 배너 없음, 마이페이지 내 고정 버튼으로 제공
    - 미설치 상태: "홈 화면에 추가" 버튼 노출 → 클릭 시 브라우저 설치 프롬프트 실행
      - 설치 완료: 버튼 미노출 (localStorage에 installed: true 기록)
      - 닫기 / 나중에: 프롬프트만 닫힘, 버튼은 그대로 유지 → 언제든 재시도 가능
    - 설치 완료 상태: 버튼 미노출

### ⚠️ Should Have (권장 구현)

- PWA 푸시 알림: 브라우저 기반 실시간 푸시 알림 (S4)

## 4. 비기능 요구사항

### 성능 (Performance)
- Next.js 15 Server Components 활용, 초기 LCP 1.5초 이내
- next/image를 통한 이미지 최적화 및 데이터 소모 최소화

### PWA & 오프라인 지원
- Service Worker 캐싱으로 네트워크 불안정 시 최근 조회한 상품 목록 열람 가능
- 오프라인 상태에서 홈 화면 진입 지원
- 캐시 제외 항목: 주문 정보, 결제 정보, 인증 토큰

### 지원 기기
- Mobile-first: iOS / Android 브라우저 최적화 반응형 디자인
- Desktop: 태블릿 및 PC 환경 가독성 유지

### 실시간 데이터
- Firestore 실시간 리스너로 주문 상태 변경을 클라이언트에 즉시 반영
- Redis, WebSocket 별도 구성 없음 (Firestore가 역할 대체)

## 6. MVP 범위 및 판매자 확장 계획

### MVP 범위 정의

```
MVP (현재): 단일 판매자 고정
  → 판매자: 디어 오키드 (개인 사업자)
  → 소비자는 디어 오키드의 상품만 탐색/주문

향후 확장: 다중 판매자 마켓플레이스 구조
  → 지역 기반 화훼/농산물 판매자 다수 입점
  → 판매자별 가게 페이지, 지역 필터링 활성화
```

---

### 단일 → 다중 판매자 전환 전략

MVP를 단일 판매자로 시작하더라도, **처음부터 storeId 구조를 DB와 API에 심어두면** 나중에 전환 비용이 크게 줄어듭니다.

#### DB 설계 원칙

```
❌ MVP에서 하지 말아야 할 설계
  products 테이블에 storeId 컬럼 없음
  → 나중에 다중 판매자 추가 시 전체 스키마 수정 필요

✅ 처음부터 이렇게 설계
  stores              { id, name, ownerId, ... }
  products            { id, storeId, name, price, saleType, ... }
  groupProductConfig  { productId, minParticipants, groupDeliveryDate, ... }
  orders              { id, storeId, userId, productId, status, ... }
  dailyCaps           { storeId, date, totalCap, usedSlots }
  deliveryFeeConfig   { storeId, directFee, hubFee, parcelFee, ... }

  MVP에서는 storeId = 'dear-orchid' 하나만 존재
  → 다중 판매자 추가 시 stores 테이블에 행만 추가하면 됨
```

#### API 설계 원칙

```
❌ MVP 단순 설계 (나중에 바꾸기 어려움)
  GET /products
  GET /orders

✅ 처음부터 storeId 포함 설계
  GET /stores/:storeId/products
  GET /stores/:storeId/orders

  MVP: /stores/dear-orchid/products 하나만 사용
  확장: /stores/new-seller/products 추가만 하면 됨
```

#### 전환 단계별 작업 목록

**Phase 1 (MVP - 단일 판매자)**
```
- stores 테이블에 디어 오키드 1개 row 생성
- 모든 상품/주문에 storeId 컬럼 포함 (값은 고정)
- 판매자 앱: 단일 판매자 전용 UI (로그인하면 바로 내 가게)
- 소비자 앱: 가게 선택 UI 없음 (단일 가게이므로)
```

**Phase 2 (다중 판매자 전환)**
```
추가 개발 항목:
  - 판매자 회원가입 / 가게 등록 기능
  - 소비자 앱: 가게 목록, 가게 상세 페이지 추가
  - 지역 기반 필터링 UI 활성화
  - 정산 시스템: 판매자별 수익 분리

변경 없는 항목 (처음부터 잘 설계했다면):
  - DB 스키마 (storeId 이미 존재)
  - API 엔드포인트 구조
  - 주문 처리 로직 핵심 부분
  - 실시간 추적, 알림 시스템
```

#### 전환 난이도 예상

| 항목 | 처음부터 storeId 설계 시 | 나중에 추가 시 |
|------|------------------------|--------------|
| DB 스키마 변경 | 불필요 | 마이그레이션 필요 |
| API 수정 | 불필요 | 전체 엔드포인트 수정 |
| 판매자 앱 확장 | 가게 등록 UI 추가만 | 권한 구조 전면 재설계 |
| 소비자 앱 확장 | 가게 목록 화면 추가만 | 상품/주문 흐름 전면 수정 |
| 예상 추가 공수 | 2~3주 | 2~3개월 |

## DB 스키마 정의 (전체)

### stores
```
stores {
  id: string               // 판매자 고유 ID
  name: string             // 가게명
  ownerId: string          // 판매자 계정 ID
  createdAt: timestamp
}
```

### products (공통)
```
products {
  id: string
  storeId: string          // stores 참조
  name: string             // 상품명
  description: string      // 상품 설명
  images: string[]         // 이미지 URL 배열
  price: number            // 기본 가격 (실시간 시세 반영)
  category: 'cut_flower' | 'orchid' | 'foliage'  // 절화 | 난 | 관엽
  colors: string[]         // 색상 배열 (멀티 선택, 판매자 등록 시 설정)
                           // 예: ["레드", "화이트"]
                           // 옵션: 레드, 핑크, 화이트, 옐로우, 오렌지, 퍼플,
                           //       블루, 그린, 무늬, 브라운, 베이지, 블랙, 그레이
  saleType: 'normal' | 'group'
  deliverySize: 'small' | 'medium' | 'large'  // 배송비 계산용 (판매자 등록 시 설정)
  isActive: boolean        // 판매 활성 여부
  createdAt: timestamp
  updatedAt: timestamp
}
```

### groupProductConfig (공동구매 전용 — saleType: 'group'일 때만)
```
groupProductConfig {
  productId: string            // products 참조 (1:1)
  minParticipants: number      // 최소 모집 인원
  maxParticipants: number      // 최대 모집 인원 (= Daily Cap)
  recruitDeadline: timestamp   // 모집 마감일 (판매자 설정)
  groupDeliveryDate: timestamp // 배송 예정일 (판매자 지정, 소비자 변경 불가)
  currentParticipants: number  // Firestore 실시간 집계
  groupDeliveryMethod: 'direct' | 'parcel'  // 배송 수단 (판매자 지정)
  deliveryFeeDiscount: number  // 할인 배송비 (0이면 무료 배송)
}
```

### orders
```
orders {
  id: string
  storeId: string              // stores 참조
  userId: string               // 로그인 필수 (비회원 미지원)
  productId: string            // products 참조
  quantity: number             // 주문 수량
  saleType: 'normal' | 'group' // 주문 시점 스냅샷

  status: 'PENDING'                        // [내부] 결제 처리 중 (Portone webhook 수신 전, 소비자 미노출, 15분 자동 삭제)
        | 'RECRUITING' | 'CONFIRMED' | 'ACCEPTED' | 'PREPARING'
        | 'DELIVERING' | 'HUB_ARRIVED' | 'PICKED_UP'
        | 'DELIVERED' | 'CANCELLED' | 'REVIEWED'

  requestedDeliveryDate: timestamp  // 소비자 선택 배송 희망일 (일반 판매)
                                    // 공동구매는 groupProductConfig.groupDeliveryDate 참조

  deliveryMethod: 'direct' | 'hub' | 'parcel'
  deliveryFee: number              // 최종 확정 배송비
  deliveryAddress: {
    address: string                // 도로명 주소
    addressDetail: string          // 상세 주소
    zipCode: string
  }
  isMetropolitan: boolean          // 서울/경기 여부 (주소 판별 결과)
  pickupCode: string | null        // 거점 픽업(hub)일 때만 생성

  totalAmount: number              // 최종 결제 금액 (상품가 + 배송비)
  cancelReason: string | null      // 취소 사유 (CANCELLED 상태 시)

  createdAt: timestamp
  updatedAt: timestamp
}
```

### dailyCaps (날짜별 배송 수용량)
```
dailyCaps {
  id: string
  storeId: string          // stores 참조
  date: string             // 'YYYY-MM-DD'
  totalCap: number         // 해당 날짜 최대 주문 수용량 (직배송 + 거점 픽업 합산)
  usedSlots: number        // 현재 사용된 슬롯 수 (Firestore 실시간 갱신)
}
```

### deliveryFeeConfig (배송비 설정 — 판매자 앱에서 관리)
```
deliveryFeeConfig {
  storeId: string               // stores 참조
  directFee: number             // 꽃차 직배송 기본 배송비 (임시: 3,000원)
  hubFee: number                // 거점 픽업 기본 배송비 (임시: 1,000원)
  parcelFee: number             // 택배 기본 배송비 (임시: 4,000원)
  freeThresholdDirect: number   // 직배송 무료 배송 기준 금액 (임시: 50,000원)
  freeThresholdHub: number      // 거점 픽업 무료 배송 기준 금액 (임시: 30,000원)
  freeThresholdParcel: number   // 택배 무료 배송 기준 금액 (임시: 50,000원)
  weatherRestrictionActive: boolean  // 기상 제한 모드 활성 여부 (판매자 수동 제어)
  updatedAt: timestamp
}
```
